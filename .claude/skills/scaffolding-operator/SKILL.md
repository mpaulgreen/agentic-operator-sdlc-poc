# Scaffolding Operator Projects

Generate complete OpenShift operator project structures without requiring the `operator-sdk` CLI. Produces all boilerplate files (Makefile, Dockerfile, cmd/main.go, config/ Kustomize overlays, PROJECT metadata) with correct conventions.

## Template Variables

Collect these from the user before generating. Derive defaults where possible.

| Variable | Description | Example | Default |
|----------|-------------|---------|---------|
| `OPERATOR_NAME` | Project name (kebab-case) | `redis-operator` | — (required) |
| `DOMAIN` | API group domain | `redis.example.com` | — (required) |
| `REPO_PATH` | Go module path | `github.com/example/redis-operator` | — (required) |
| `API_GROUP` | Short API group name | `cache` | — (required) |
| `KIND` | CRD kind (PascalCase) | `RedisCluster` | — (required) |
| `API_VERSION` | API version | `v1alpha1` | `v1alpha1` |
| `GO_VERSION` | Go version | `1.22` | `1.22` |
| `CONTAINER_TOOL` | Build tool | `podman` | `podman` |
| `IMAGE_TAG_BASE` | Image registry path | `quay.io/example/redis-operator` | `DOMAIN/OPERATOR_NAME` |
| `NAMESPACED` | Whether the CRD is namespace-scoped | `true` | `true` |

Derived variables (do not ask user):
- `KIND_LOWER` = lowercase of KIND (e.g., `rediscluster`)
- `KIND_PLURAL` = pluralized form of KIND_LOWER following Kubernetes conventions:
  - Ends in `s`, `x`, `z`: no change (e.g., `redis` → `redis`, `clusterredis` → `clusterredis`)
  - Ends in `ch`, `sh`: add `es` (e.g., `statusmatch` → `statusmatches`)
  - Ends in `y` preceded by consonant: replace `y` with `ies` (e.g., `alertpolicy` → `alertpolicies`)
  - Otherwise: add `s` (e.g., `rediscluster` → `redisclusters`)
  - User can override with `--plural` if the auto-derived form is wrong
- `LEADER_ELECTION_ID` = random 8-char hex + `.` + DOMAIN (e.g., `a1b2c3d4.redis.example.com`)
- `YEAR` = current year (e.g., `2026`) — used in license headers

## Workflow A: New Operator Project

Use when the user asks to create a new operator from scratch.

1. Collect template variables from the user (name, domain, repo, group, kind).
2. Create the project directory structure. See `references/project-layout.md` for the full layout.
3. Generate files using templates from `assets/templates/`:
   - `PROJECT` from `project.tmpl`
   - `cmd/main.go` from `main.go.tmpl`
   - `Makefile` from `makefile.tmpl`
   - `Dockerfile` from `dockerfile.tmpl`
   - `api/<version>/groupversion_info.go` from `groupversion_info.go.tmpl`
   - `api/<version>/doc.go` — package doc with generator markers
   - `.gitignore` from `gitignore.tmpl`
   - `.dockerignore` from `dockerignore.tmpl`
   - `.golangci.yml` from `golangci-lint.yml.tmpl`
   - `README.md` from `readme.md.tmpl`
   - `go.mod` — module declaration with Go version and controller-runtime dependency
   - `hack/boilerplate.go.txt` — Apache 2.0 license header (referenced by `make generate` via `controller-gen object:headerFile`; without it `make generate`, `make build`, and `make test` all fail)
4. Generate config/ Kustomize files. See `references/kustomize-structure.md` for all files and their contents.
5. Generate placeholder files (prepend license header from `hack/boilerplate.go.txt` to all generated `.go` files):
   - `api/<version>/<kind_lower>_types.go` — empty type stubs (Spec, Status, root type, list type) with TODO markers. If `NAMESPACED=false`, add `//+kubebuilder:resource:scope=Cluster` marker on the root type and omit `namespaced: true` from the PROJECT resource entry.
   - `internal/controller/<kind_lower>_controller.go` — empty controller stub with Reconcile() returning empty result
6. Generate `api/<version>/zz_generated.deepcopy.go` with DeepCopyInto, DeepCopy, and DeepCopyObject methods for all types (root type, list type, spec, status). The types register with `SchemeBuilder.Register()` which requires `runtime.Object` interface — without DeepCopy methods the project will not compile. This file is later regenerated by `make generate` when types are updated.
7. Run `go mod tidy` to resolve all dependencies.
8. Validate the result using `scripts/validate-project-structure.sh`.

## Workflow B: Add Resource to Existing Project

Use when the user asks to add a new resource/kind to an existing operator.

1. Collect: new API_GROUP, KIND, API_VERSION, NAMESPACED (default true).
2. Read the existing `PROJECT` file. Check if the new API_GROUP differs from existing groups.
3. **Determine layout mode:**
   - **Same group** (e.g., adding `RedisSentinel` to existing group `cache`): use flat layout — files go in existing `api/<version>/` and `internal/controller/`.
   - **Different group** (e.g., adding `monitoring` when `cache` exists): use **multi-group layout** — each group gets its own subdirectory.
4. Append a new resource entry to the PROJECT file's `resources:` list.
5. **If multi-group (different group):**
   - Add `multigroup: true` to PROJECT (if not already set).
   - Create `api/<group>/<version>/` directory.
   - Create `internal/controller/<group>/` directory.
   - Generate in the new directories:
     - `api/<group>/<version>/groupversion_info.go` — new SchemeBuilder for this group
     - `api/<group>/<version>/<kind_lower>_types.go` — empty type stubs (add `//+kubebuilder:resource:scope=Cluster` if `NAMESPACED=false`)
     - `api/<group>/<version>/zz_generated.deepcopy.go` — DeepCopy methods for new types
     - `internal/controller/<group>/<kind_lower>_controller.go` — controller stub in its own package
   - Update `cmd/main.go` with **aliased imports**:
     - `<group><version> "<repo>/api/<group>/<version>"` (e.g., `monitoringv1alpha1`)
     - `<group>controller "<repo>/internal/controller/<group>"` (e.g., `monitoringcontroller`)
     - Add scheme registration: `utilruntime.Must(<group><version>.AddToScheme(scheme))`
     - Add controller setup: `<group>controller.<Kind>Reconciler{...}.SetupWithManager(mgr)`
6. **If same group:**
   - Generate in existing directories:
     - `api/<version>/<kind_lower>_types.go` — empty type stubs (add `//+kubebuilder:resource:scope=Cluster` if `NAMESPACED=false`)
     - `internal/controller/<kind_lower>_controller.go` — controller stub
   - Update `api/<version>/zz_generated.deepcopy.go` — append DeepCopy methods for new types.
   - Update `cmd/main.go`:
     - Add controller setup block (no new imports needed — same package)
7. **In both cases, also:**
   - Generate `config/rbac/<group>_<kind_lower>_editor_role.yaml` — ClusterRole for end users to edit CRs.
   - Generate `config/rbac/<group>_<kind_lower>_viewer_role.yaml` — ClusterRole for end users to view CRs.
   - Append the editor/viewer role entries to `config/rbac/kustomization.yaml`.
   - Generate `config/samples/<group>_<version>_<kind_lower>.yaml` — sample CR.
   - Append the sample entry to `config/samples/kustomization.yaml`.
   - Append CRD base reference to `config/crd/kustomization.yaml`: `bases/<group>.<domain>_<kind_plural>.yaml`.
8. Do NOT regenerate Makefile, Dockerfile, or top-level config/ files — they are shared.
